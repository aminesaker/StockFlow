# Internationalisation (i18n) — TijaraFlow

Architecture multilingue de TijaraFlow. Langues : **français (`fr`, défaut)** et **anglais (`en`)**.
Stack : **next-intl v4** sur Next.js 16 (App Router), stratégie **cookie + détection navigateur**.

---

## 1. Vue d'ensemble

| Élément | Rôle |
|---|---|
| `i18n/locales.ts` | Liste des langues, langue par défaut, helpers (`isLocale`, `pickLocale`). |
| `i18n/request.ts` | Config par requête : lit le cookie `NEXT_LOCALE`, sinon `Accept-Language`, charge `messages/<locale>.json`. |
| `next.config.ts` | Branche le plugin `next-intl` (`createNextIntlPlugin('./i18n/request.ts')`). |
| `middleware.ts` | Pose le cookie `NEXT_LOCALE` au premier passage (détection `Accept-Language`) + garde d'authentification. |
| `app/layout.tsx` | `NextIntlClientProvider` (locale + messages) ; `<html lang={locale}>`. |
| `messages/fr.json`, `messages/en.json` | Catalogues de traduction centralisés. |
| `components/i18n/LanguageSwitcher.tsx` | Sélecteur de langue (cookie + persistance `user_settings.locale`). |
| `app/actions/locale.ts` | Server action : écrit le cookie et persiste la langue de l'utilisateur connecté. |

### Comment la langue est choisie (ordre de priorité)
1. Cookie `NEXT_LOCALE` (choix explicite via le switcher).
2. En-tête `Accept-Language` du navigateur (premier passage).
3. Langue par défaut : **français**.

Le choix est durable (cookie 1 an) et, pour les comptes connectés, persisté en base (`user_settings.locale`) — utile pour l'envoi d'e-mails dans la bonne langue.

## 2. Utilisation dans le code

### Composant serveur (RSC)
```tsx
import { getTranslations } from 'next-intl/server'

export default async function Page() {
  const t = await getTranslations('nav')
  return <a>{t('pricing')}</a>
}
```

### Composant client
```tsx
'use client'
import { useTranslations } from 'next-intl'

export default function Btn() {
  const t = useTranslations('auth.login')
  return <button>{t('submit')}</button>
}
```

### Interpolation et tableaux
```tsx
t('register.successBody', { email })       // « envoyé à … »
const items = t.raw('faq.items')           // tableau JSON brut (objets/listes)
```

### Métadonnées SEO localisées
```tsx
export async function generateMetadata() {
  const t = await getTranslations('landing.meta')
  return { title: t('title'), description: t('description') }
}
```

### Règle d'or
**Aucun texte visible ne doit être codé en dur.** Toute chaîne destinée à l'utilisateur passe par une clé de `messages/*.json`.

## 3. Ajouter / modifier une traduction

1. Ajouter la clé dans `messages/fr.json` **et** `messages/en.json` (mêmes clés dans les deux fichiers).
2. Référencer la clé via `t('namespace.key')`.
3. Vérifier la validité JSON :
   ```bash
   node -e "JSON.parse(require('fs').readFileSync('messages/fr.json'));JSON.parse(require('fs').readFileSync('messages/en.json'))"
   ```

**Convention de nommage** : namespaces par domaine (`nav`, `auth`, `landing`, `pricing`, `features`, `contact`, …), clés en `camelCase`, sous-objets pour les sections.

## 4. Bonnes pratiques SEO multilingue

L'architecture actuelle (cookie, URL unique) est sûre et immédiate. Pour un référencement multilingue optimal, appliquer :

- **Balises `hreflang` / `alternates`** sur les pages publiques, via `generateMetadata` :
  ```tsx
  export async function generateMetadata() {
    return {
      alternates: {
        languages: {
          fr: 'https://app.tijaraflow.com/',
          en: 'https://app.tijaraflow.com/en',
          'x-default': 'https://app.tijaraflow.com/',
        },
      },
    }
  }
  ```
- **`<html lang>`** déjà géré dynamiquement par la langue active.
- **Titres et descriptions localisés** (déjà en place sur landing, tarifs, fonctionnalités, contact).
- **Sitemap** : exposer une entrée par langue (`/sitemap.xml`) avec les alternances `hreflang`.
- **Canonical** : une URL canonique par page pour éviter le contenu dupliqué.
- **Contenu réellement traduit** (pas de traduction automatique côté client), ce qui est le cas ici.

### Recommandation pour un SEO maximal : URLs localisées
Le SEO multilingue est le plus performant avec **une URL par langue** (`/fr/...`, `/en/...`).
C'est l'évolution prévue (voir §6). En attendant, les balises `hreflang` + contenu traduit
couvrent l'essentiel pour un démarrage bilingue.

> Note : le dashboard est derrière authentification (non indexé) — le SEO ne le concerne pas.
> L'effort SEO porte uniquement sur les pages publiques.

## 5. Localisation des e-mails

- La langue de l'utilisateur est persistée dans `user_settings.locale`.
- Les fonctions d'envoi (`lib/email/send.ts`) acceptent une `locale` et sélectionnent les
  variantes FR/EN des gabarits.
- Côté serveur (crons, webhooks, actions), lire `user_settings.locale` du destinataire avant l'envoi.

## 6. Ajouter une nouvelle langue (Espagnol, Allemand, Arabe…)

L'architecture est conçue pour être **scalable**. Pour ajouter une langue (ex. `es`) :

1. **Déclarer la langue** dans `i18n/locales.ts` :
   ```ts
   export const locales = ['fr', 'en', 'es'] as const
   export const localeNames = { fr: 'Français', en: 'English', es: 'Español' }
   export const localeFlags  = { fr: '🇫🇷', en: '🇬🇧', es: '🇪🇸' }
   ```
2. **Créer le catalogue** `messages/es.json` (copier `fr.json`, traduire les valeurs — mêmes clés).
3. C'est tout : le switcher, la détection navigateur et le chargement des messages s'adaptent automatiquement.

### Cas particulier : langues RTL (arabe `ar`, hébreu `he`)
Les langues écrites de droite à gauche nécessitent `dir="rtl"` :

1. Ajouter un helper dans `i18n/locales.ts` :
   ```ts
   export const rtlLocales = ['ar', 'he'] as const
   export function isRtl(locale: Locale) {
     return (rtlLocales as readonly string[]).includes(locale)
   }
   ```
2. Appliquer la direction dans `app/layout.tsx` :
   ```tsx
   const locale = await getLocale()
   return <html lang={locale} dir={isRtl(locale) ? 'rtl' : 'ltr'}>…</html>
   ```
3. Préférer les utilitaires **logiques** de Tailwind (`ms-`, `me-`, `ps-`, `pe-`, `text-start`,
   `text-end`) plutôt que `ml-`, `mr-`, `text-left`… afin que la mise en page se reflète
   automatiquement en RTL.
4. Vérifier les icônes directionnelles (flèches) et les graphiques.

### Tableau récapitulatif
| Langue | Code | Sens | Fichier à créer | Spécificité |
|---|---|---|---|---|
| Espagnol | `es` | LTR | `messages/es.json` | — |
| Allemand | `de` | LTR | `messages/de.json` | Textes plus longs (prévoir l'espace UI). |
| Arabe | `ar` | **RTL** | `messages/ar.json` | `dir="rtl"` + utilitaires logiques. |

## 7. Liste des textes traduits (couverture actuelle)

**222 clés** traduites par langue (FR + EN), réparties par namespace :

| Namespace | Clés | Surface |
|---|---|---|
| `common` | 5 | éléments transverses (email, mot de passe, retour…) |
| `nav` | 5 | navigation publique (header) |
| `footer` | 5 | pied de page public |
| `auth` | 32 | connexion, inscription, mot de passe oublié, nouveau mot de passe |
| `landing` | 78 | page d'accueil (hero, problèmes, solution, étapes, valeurs, FAQ, démo, maquette) |
| `demoForm` | 14 | formulaire de demande de démo |
| `pricing` | 29 | page Tarifs (forfaits, points clés, CTA) |
| `features` | 27 | page Fonctionnalités (9 cartes + CTA) |
| `contact` | 11 | page Contact (titres, blocs d'info) |
| `contactForm` | 16 | formulaire de contact |

### Surfaces migrées (FR/EN, en production)
- Header & footer publics (+ Language Switcher)
- Pages d'authentification : connexion, inscription, réinitialisation (demande + mise à jour)
- Pages publiques : accueil, tarifs, fonctionnalités, contact
- Formulaires : démo, contact

### Couverture complète (FR/EN, en production)
- **Dashboard intégral** : vue d'ensemble, stocks, mouvements, prévisions, commandes
  (liste + détail + actions), clients, factures (liste + détail + actions), intégrations,
  automatisations, rapports, facturation, paramètres (notifications, clés API, WooCommerce),
  onboarding, documentation API.
- **Pages de paiement** (succès / annulation) et **PDF de facture client** (locale du marchand).
- **E-mails** : gabarits FR/EN complets.
- **Toasts, validations, confirmations** des formulaires dashboard.

Chaque module suit le même patron : extraire les chaînes vers un namespace dédié, puis
remplacer par `t('module.key')`. Les composants rendus hors du provider next-intl
(PDF via `@react-pdf/renderer`, e-mails) reçoivent une `locale` explicite avec un
dictionnaire FR/EN, à l'image de `lib/email`.
